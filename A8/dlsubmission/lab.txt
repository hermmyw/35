Which of these calls are related to dynamic linking and what is the relationship?
(1) access("/etc/ld.so.preload", R_OK)      = -1 ENOENT (No such file or directory)
/etc/ld/so.preload replaces LD_PRELOAD, which is a list of additional, user-specified, ELF share 
libraries that should be loaded before others. 
(2) open("/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3
/etc/ld.so.cache is a cache map of plain dynamic library names.
(3) open("/lib64/libc.so.6", O_RDONLY|O_CLOEXEC) = 3
/lib64/libc.so.6 is the actual dynamic library linked to the program.

5. Invoke ldd on each command in your list. If there are error messages, investigate why they're occurring.
Shell script to see the dynamic libraries:
	#!/bin/bash
	#ldd_run
	for x in "$(ls /usr/bin | awk \
	'NR%101==704978214%101' $1)"; do
	  y=`which $x`
	  ldd $y
	done

$ ldd gunzip
	not a dynamic executable
$ ldd unicode_start 
	not a dynamic executable
$ ldd xzmore 
	not a dynamic executable

The errors might happen due to the following reasons:
(1) The program is a shell script and does not depend on any shared libraries.
(2) ldd does not work on a.out shared libraries.
(3) ldd  does  not  work  with  some extremely old a.out programs which were built
before ldd support was added to the compiler releases. 

The three files with errors are Shell scripts. Shell scripts are interpreted directly 
by the Shell instead of by the CPU. A Shell script does not link to any libraries.
